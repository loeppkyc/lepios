## TL;DR

- **~1,252 LOC of safety code already exists** across `lib/safety/checker.ts` (411), `lib/harness/safety/{static,index,llm-review,approval}.ts` (841), plus `scripts/pre-commit-safety.mjs` and `scripts/verify-safety.ts`.
- **5 of 6 signals have a working static/regex base.** What's missing is the **risk-score weighting**, the **PR-level diff scope** (existing code runs at pre-commit on staged diff, not against PR-vs-base), and the **deploy-gate integration** (Safety today blocks at pre-commit; T-002 wants it scoring at the gate).
- **Test coverage delta** has no foundation (no coverage-v8 dep, no baseline). Needs the most net-new work.
- **Puppeteer E2E** has primitives (Puppeteer 24, `vitest.e2e.config.ts`, one acceptance spec, `arms-legs/browser-handlers.ts`) but no regression harness on PR/preview.
- **Realistic wiring estimate to reach T-002 done_state: ~600–800 LOC of glue code** + ~120 LOC of new schema/migration. **Rebuild estimate would be ~2,000+ LOC.**
- **The user's prompt mis-cites F-N9 as an "800-LOC threshold."** F-N9 is the Vercel cron-count incident (max 18 crons on Hobby). No 800-LOC enforcement exists in the codebase — flag for clarification before Phase 2.

## Deploy gate position + Safety Agent slot

**Today's gate flow** (`app/api/cron/deploy-gate-runner/route.ts`):

1. Cron polls `agent_events` for `task_type='deploy_gate_triggered'` rows (status='success' = tests_passed=true)
2. For each pending commit_sha:
   - Write `deploy_gate_processing` marker (60s window prevents double-claim)
   - Poll Vercel for preview deployment by commit_sha
   - On `ready`: run `runSmokeCheck(preview_url)` — single HTTP probe to `/api/health`
   - If smoke passes: `runSchemaCheck(commit_sha, branch)` — detect migration files via GitHub API
   - If schema-clean: `runAutoPromote` (merge to main, send promotion notification, queue post-merge production smoke)
   - If schema-migrations: `runMigrationGate` — fetch SQL, send Telegram approval card
3. Post-merge: `runProductionSmokes` runs `route-health` + `cron-registration` + `ollama-health` against production

**Where T-002 Safety Agent slots in:** Between step 2's smoke-pass and step 2's schema check. The gate already has `commit_sha`, `branch`, `task_id`, `preview_url` in scope — same context Safety needs. Concretely:

```
preview ready → smoke pass → [Safety Agent: compute risk score] → schema check
                                  ↓
                          Low (<30): proceed
                          Med (30–70): twin arbiter → PROCEED/HOLD/ESCALATE
                          High (>70): Telegram Colin, do not promote
```

**Why this slot, not pre-commit:**

- Pre-commit already does Layer 0 (static safety block on staged diff). That's a different decision: "should this commit exist?" Layer 0 is already wired and working.
- T-002's risk score needs PR-vs-base diff (LOC, coverage delta, all-touched-files signature) — only available at the gate, not at commit time.
- The auto-merge decision is the value-add. Pre-commit can't auto-merge; the gate already does.

**What does NOT need to move:**

- Pre-commit Layer 0 (`scripts/pre-commit-safety.mjs`) keeps its current scope as a fast block. Even if Safety Agent runs at the gate, the commit-time block stays — defense in depth.
- `lib/safety/checker.ts` and `lib/harness/safety/static.ts` are reusable as-is. The gate adapter calls `staticSafetyCheck({ diff, sql })` with PR diff (vs base), takes findings, maps to score.

## Risks / open questions for Phase 2 confirmation

1. **F-N9 mis-citation in prompt.** Prompt says "coordinator already enforces 800 LOC threshold per F-N9 — is that wired?" F-N9 is the Vercel-cron-count incident. No LOC threshold exists in the codebase. Confirm whether (a) plan_loc enforcement is genuinely new work, or (b) there's a different rule the prompt was reaching for.
2. **PR-vs-base diff visibility from cron context.** The gate runs as a Vercel cron with no checked-out git. Today's deploy gate fetches migration SQL via the **GitHub API** (`fetchMigrationSQL`). Same path will be needed for the full PR diff — likely a new helper. Pre-commit code paths can't be reused 1:1.
3. **Risk-score weights and thresholds** are spec'd but not stored anywhere. T-002 says `harness_config` keys (`SAFETY_WEIGHT_*`, `SAFETY_THRESHOLD_*`). Migration to seed those.
4. **Twin arbiter dependency.** Twin Q&A is at 68% per inventory; vector layer at 0% (FTS-only). Twin needs to handle PR-context queries reliably before the medium-risk path activates. T-002 calls this out: "may need a sub-task to harden twin first."
5. **Coverage delta cost.** Running coverage twice per PR (base + head) adds ~10 min CI time. A cached-base-coverage approach (compute on each main commit, store JSON in `harness_config` or a new table) avoids the double-run. Choose before Phase 2.
6. **Puppeteer's role in T-002.** Spec table doesn't list Puppeteer as a score input. Treat as separate post-merge regression workstream, or fold into the medium-risk twin context as "regression evidence"? Default reading: separate post-merge regression, replacing/augmenting the current 5-route status-code smoke.

| Puppeteer E2E (post-merge regression, 5 pages)      | arms-legs/browser-handlers.ts        | ~150                   |
| Risk scorer (signals → 0–100)                       | —                                    | ~80                    |
| Twin arbiter route + comparable-decisions retrieval | lib/twin/\*                          | ~120                   |
| Decision router + HOLD-retry-24h                    | task_queue                           | ~80                    |
| Daily summary digest + dashboard counter            | morning_digest, /autonomous          | ~60                    |
| **Total**                                           | **~1,250 LOC reused**                | **~990 + 1 migration** |

Rebuilding from scratch would be ~2,500 LOC. **Building on the existing safety/ codebase saves ~60% of the work.** The lift is mostly glue + coverage tooling + signature-matching read path, not new analyzers.

**Phase 2 should NOT touch:** the pre-commit Layer 0 path, `lib/safety/checker.ts` rule logic, or `scripts/verify-safety.ts`. They work and are already wired.

**Phase 2 should:** add a deploy-gate-runner slot that fetches PR diff via GitHub API, calls existing checkers + new signature-matcher + new coverage-differ, sums to a risk score, and routes to auto-merge / twin / Colin per the spec.
