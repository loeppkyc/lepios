## Findings

### CRITICAL — fixed this audit

**F-E1: Chat route hits localhost in production**

- `app/api/chat/route.ts:16` and `lib/orb/ollama.ts:37` both read `OLLAMA_BASE_URL` with a `http://127.0.0.1:11434` fallback
- `OLLAMA_BASE_URL` was **NOT set** in Vercel
- Result: chat POST would call `127.0.0.1:11434/api` from a Vercel serverless container — connection refused
- The "production tunnel verified" tracker entries (A1/B5/C2/C6) refer to the embed/twin path via `lib/ollama/client.ts` which correctly uses `OLLAMA_TUNNEL_URL`. The chat path was never actually exercised in production.
- **Fix applied:** Set `OLLAMA_BASE_URL=https://ollama.loeppky.xyz` in Vercel production (F15-clean: `printf '%s'` + `cat -A` verified).
- **Follow-up:** consolidate chat route + `lib/orb/ollama.ts` to read `OLLAMA_TUNNEL_URL` directly so there's one canonical env var.

**F-E2: UI label drift**

- `app/(cockpit)/chat/page.tsx:9` reads `NEXT_PUBLIC_OLLAMA_CHAT_MODEL` (default `qwen2.5-coder:3b`); displays "running on {CHAT_MODEL}" at line 395
- Server-side `OLLAMA_CHAT_MODEL` was just set to `qwen2.5:7b`, but the client `NEXT_PUBLIC_*` was unset → UI showed wrong model
- **Fix applied:** Set `NEXT_PUBLIC_OLLAMA_CHAT_MODEL=qwen2.5:7b` in Vercel.
- **Follow-up:** UI should fetch model from server, not from a separate `NEXT_PUBLIC_*` env, to eliminate drift permanently.

### MEDIUM — flagged, not fixed

Used in 7 files (production paths):

- `app/api/expenses/import/route.ts` — expense import classification
- `app/api/receipts/scan/route.ts` — receipt OCR / extraction
- `lib/twin/query.ts` — frontier fallback when local Ollama unavailable
- `lib/harness/self-repair/drafter.ts` — autonomous drafter
- `lib/purpose-review/summary.ts` — purpose review summarisation
- `scripts/ai-review.mjs` — pre-commit AI review (already tracked: task `efa60e5c`)
- `scripts/measure-handoff-tokens.ts` — utility

**Decision required:** is Anthropic frontier on or off in production?  
- If on: add `ANTHROPIC_API_KEY` to Vercel.  
- If off (Ollama-only architecture): wrap each call site in a `if (!apiKey) return null` guard so failure mode is explicit, not a 500.

**F-E4: LEPIOS_BASE_URL not set**

- Used in `app/api/cron/deploy-gate-runner/route.ts:398` with `?? 'https://lepios-one.vercel.app'` fallback. Functions today, but explicit is better than implicit.
- Already tracked: task `b93658c2` ("Fix coordinator drain_trigger_failed — add LEPIOS_BASE_URL to harness_config"). Recommendation: add to `harness_config` (DB-resident config pattern), not Vercel env (S-L1).

### LOW — drift risk, monitor

**F-E5: CRON_SECRET duplicated in Vercel + harness_config**

- Vercel: process.env.CRON_SECRET (used by `lib/auth/cron-secret.ts`)
- harness_config: `CRON_SECRET` row (used by autonomous agents per S-L1 pattern)
- Both currently have the same value (64 chars; verified functionally — `/api/cron/oura-sync` accepted the harness_config value via the `requireCronSecret` helper which reads `process.env`)
- **Risk:** rotation drift. If one is rotated and the other isn't, autonomous agents will diverge from cron auth.
- **Recommendation:** add a daily `harness_config` integrity check that asserts `CRON_SECRET` matches the Vercel env value (via internal API call), alerts on mismatch.

**F-E6: TELEGRAM_CHAT_ID duplicated similarly** — same as F-E5; same recommendation.

**F-E7: Several Ollama model env vars unset, defaults silently apply**

- `OLLAMA_TWIN_MODEL` defaults to `qwen2.5:32b` (model exists ✓ — verified on tunnel)
- `OLLAMA_ANALYSIS_MODEL`, `OLLAMA_CODE_MODEL`, `OLLAMA_EMBED_MODEL`, `OLLAMA_GENERAL_MODEL` — defaults in code, not in Vercel
- These defaults are sensible but should be set explicitly in Vercel once we know which models we're standardising on, so the source of truth is one place.
- Not urgent; flag for next sprint.

**F-E8: Sandbox / exec env vars unset**

- `EXEC_SECRET`, `SANDBOX_EXEC_SECRET`, `SANDBOX_EXEC_URL`, `SANDBOX_SPIKE_ENABLED`
- Used by `scripts/sandbox-spike.ts` and `scripts/exec-server.js` — local dev only, not production paths
- Not a gap.
