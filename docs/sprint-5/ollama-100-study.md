## What's Missing (the 90%)

### Gap 1 — 60-second timeout making every outage a 60-second tax (CRITICAL)

**Production evidence:**
Every `ollama.generate` failure in production has `duration_ms: 60006-60073ms`. The client
is configured with a 60-second fetch timeout. When Ollama is down (which is most of the
time per production data), every Twin ask burns 60 seconds before the Claude fallback fires.

**Real cost:** Total `twin.ask` duration when Ollama is down = 62-70 seconds. With a 15-second
timeout, that drops to ~17-20 seconds. Same result, 3.5× faster.

**Fix:** Reduce generate timeout from 60s → 15s in `lib/ollama/client.ts`. 15 seconds is
generous for a local model over a tunnel. Any model that hasn't responded in 15s is either
loading (cold start) or dead.

---

### Gap 2 — No circuit breaker: every call retries a known-dead Ollama (CRITICAL)

**Production evidence:** In the session starting at 13:17 UTC, 9 consecutive `ollama.generate`
failures (all 60s timeouts) were made by the coordinator's Twin phase. Each call independently
attempted Ollama. Total wasted time: 9 × 60s = 9 minutes of waiting for an Ollama that was
provably dead after call #1.

With a circuit breaker: after 3 consecutive failures within 5 minutes, subsequent calls skip
Ollama immediately. Ollama-down sessions go from 9-minute stalls to instant fallbacks.

**Approach for this codebase:**
A full in-memory circuit breaker doesn't work in Vercel serverless functions (no shared memory
between invocations). The right approach: read the last N `agent_events` rows before each
`generate()` call to determine circuit state. One fast Supabase SELECT (~10-15ms) before each
call saves a 15-60s timeout on dead Ollama invocations.

State logic (in `lib/ollama/circuit.ts`, new):

```
CLOSED  → proceed normally
  Transition: ≥3 failures in last 5 minutes → OPEN
OPEN    → skip Ollama, return OllamaUnreachableError immediately
  Transition: last failure was >5 minutes ago → HALF_OPEN (probe)
HALF_OPEN → call healthCheck() (3s timeout) as probe
  On success → CLOSED
  On failure → OPEN
```

State is read from `agent_events` (existing table, no migration). The circuit check adds
~15ms per call but saves 15-60s per call when Ollama is down.

### Gap 3 — Failure events log `meta: null` (BUG — HIGH)

**Production evidence:** Every `ollama.generate` failure event has `meta: null`:

```json
{ "action": "ollama.generate", "status": "failure", "meta": null, "duration_ms": 5 }
```

Success events log `meta: {model, task}`. The error path in `client.ts` drops this metadata.

This means when Colin asks "which model was failing?", the answer is lost. The circuit check
also reads these events — it needs to tell `generate` calls from `embed` calls.

**Fix:** Add model and error reason to the `meta` field in the `logEvent` call in the
`catch` block of `generate()` and `embed()`:

```typescript
void logEvent('ollama', 'ollama.generate', {
  status: 'failure',
  durationMs: Date.now() - start,
  meta: { model, task, error: err instanceof Error ? err.message : String(err) },
})
```

### Gap 4 — No Telegram alert on Ollama outage (HIGH)

**Production evidence:** `metrics.digest` topErrors shows `OllamaUnreachableError count=11`
but Colin was never alerted. He discovered it by reading logs. 11 errors accumulated silently.

With an alert: Colin gets one Telegram message when Ollama transitions to OPEN state (3+
consecutive failures). He can then restart Ollama or acknowledge the tunnel is down.
One message per outage transition, not one per call.

**Implementation:** When the circuit transitions `CLOSED → OPEN`, fire a Telegram notification
via `outbound_notifications`. Reuse the existing notification path. Log
`agent_events: action='ollama.circuit_open'`.

When circuit transitions `OPEN → CLOSED` (recovery), send a recovery notification.

### Gap 5 — F17/F18: routing decision not structured; no daily rollup (MEDIUM)

**Current state:** `twin.ask` events log `ollama_unreachable: true` in meta. This is a
start but it's a boolean flag, not a structured routing decision. The morning_digest does
NOT include Ollama stats.

**What's missing:**

- F17: `routing_decision: 'ollama' | 'claude_fallback'` on every `twin.ask` event — structured
  enum that the path engine can use as a state variable ("Colin is reachable on Ollama 32%
  of the time; this is a home-presence signal")
- F18: daily rollup in morning_digest — "Ollama: N calls, M% uptime, avg latency Xms,
  fallback rate Y%"
- F18 benchmark: target ≥50% of `twin.ask` calls served by Ollama once tunnel is reliably
  configured (baseline today: 0%)

### Gap 6 — `OLLAMA_TUNNEL_URL` not validated on startup (MEDIUM)

**Current state:** If `OLLAMA_TUNNEL_URL` is unset in Vercel env, the client silently hits
`http://localhost:11434` — which will always fail in production. No warning is logged.

**Evidence from design doc (`docs/harness-step-6.5-ollama-daytime-tick.md:20-23`):**
"Until OLLAMA_TUNNEL_URL is set... every Ollama call in production silently hits nothing."

**Fix:** In `lib/ollama/client.ts` `getBaseUrl()`, if the result is localhost and NODE_ENV
is production, log one `agent_events` warning: `action='ollama.config_warning'`,
`meta: {reason: 'OLLAMA_TUNNEL_URL not set, using localhost fallback'}`. One log per
cold start (use a module-level flag).

## Production Pattern Summary

| Metric                                     | Value                  | Evidence                                           |
| ------------------------------------------ | ---------------------- | -------------------------------------------------- |
| `ollama.generate` failure rate             | 100% in recent history | All 20+ generate events in query: status='failure' |
| Generate timeout duration                  | 60,006-60,073ms        | `duration_ms` on all timeout failures              |
| `nomic-embed-text` success rate            | ~100%                  | 8/8 embed events: status='success', 78-711ms       |
| `twin.ask` served by Ollama                | 0%                     | All twin.ask events: `model='claude-sonnet-4-6'`   |
| Total twin ask latency (Ollama down)       | 62-70 seconds          | `duration_ms` on twin.ask success events           |
| OllamaUnreachableError count (last digest) | 11                     | `metrics.digest` topErrors                         |
| Failures with `meta: null`                 | 100%                   | All failure events                                 |
| Consecutive failures before fallback       | 1 (no circuit breaker) | Each call independently tries                      |

**Root cause of 100% failure rate:** `nomic-embed-text` succeeds → tunnel is reachable.
`qwen2.5:32b` always fails → the large generate model is not loaded (machine asleep or model
not running). Embedding runs on a small always-on model; generation requires a large model
that needs explicit startup.

## Phase 1b — Digital Twin Q&A

All design decisions are pattern-matchable. No Colin escalation needed.

**Resolved by twin:**

- Q: Generate timeout — 60s → 15s. Is 15s right? → Yes. 15s = generous for a local tunnel.
  Most models respond in <5s when running. A 15s timeout still catches slow cold-loads but
  doesn't keep callers waiting 60s for a dead endpoint.
- Q: Circuit state persistence — in-memory vs Supabase? → Supabase (agent_events read).
  Vercel serverless functions have no shared memory. agent_events already tracks failures.
  A single `SELECT COUNT(*) WHERE action='ollama.generate' AND status='failure' AND occurred_at > now() - interval '5 minutes'` is the circuit state. No new table needed.
- Q: Telegram alert — per failure or per transition? → Per transition (CLOSED→OPEN).
  One alert per outage. Recovery alert when circuit closes. Not one alert per call.
- Q: Daytime tick — in scope for this doc? → No. Daytime tick requires reliable Ollama,
  which is what this doc builds. Daytime tick is a follow-up chunk.
- Q: Model registry refactor — in scope? → Minimal. Keep `autoSelectModel()` as-is but
  add a `lib/ollama/models.ts` constants file so models are centralized. Not a behavior change.

## Phase 1c — Consolidated pending_colin_qs

**Q1 (confirmation, non-blocking):**
The study shows `qwen2.5:32b` has a 0% success rate in production. Is this because
(a) the Cloudflare tunnel is up but the model isn't running, or (b) the tunnel itself is
down? The `nomic-embed-text` success rate (embed always works) suggests (a). Confirming
this shapes whether the circuit breaker probe should use `/api/tags` (is server up?) or
`/api/generate` with a tiny test prompt (is this model loaded?).

My recommendation (from twin): use `/api/tags` for the probe — it's fast and tells us if
Ollama is reachable. If reachable but generate still fails, that's a model-not-loaded error
that will surface in the next actual generate call. Acceptable.

**Q2 (scope confirmation):**
Daytime tick (Step 6.5) is deferred from this acceptance doc. Confirm that "Ollama component
→ 100%" means the reliability layer (circuit breaker, liveness gate, alerting, F17/F18) is
the target, not the full Step 6.5 daytime tick.
