## Phase 1b Answers (Digital Twin)

### Q1: Embedding scope

**Answer: embed .py source (excluding tests) + .md docs. Exclude config/.toml/.json.**

Reasoning:

- .py source contains all logic, docstrings, inline comments — captured at function-level chunking
- .md docs (ARCHITECTURE.md, CLAUDE.md, CODEBASE_INDEX.md, KNOWLEDGE_SYSTEM.md,
  SYSTEM_INTEGRITY_CHECKLIST.md) contain architectural decisions and domain rules that
  answer "why" questions — the most common Twin escalation type
- Test files: EXCLUDED — mock-heavy, assertions don't represent domain logic, pollute RAG
  retrieval with patterns like `mock_sheets.return_value = ...`
- Config files (.toml, .json, .txt): EXCLUDED — low semantic value for the Twin
- Embedding candidates: ~198 .py files + 7 .md files ≈ 205 total documents

### Q2: Tier assignment heuristic

**Answer: apply the rule as primary pass; builder allowed ±1 tier based on module-specific complexity.**

Rule:

- **Tier 1** — zero Streamlit imports, pure logic/math/data (keepa_api.py, sourcing.py, fba_fees.py)
- **Tier 2** — reads `st.secrets`/`st.cache_data` but zero UI widgets (most API clients: sheets.py, amazon.py, gmail.py)
- **Tier 3** — `st.*` display calls (metrics, charts, text) but minimal session_state (simple display pages)
- **Tier 4** — significant session_state, buttons, forms (most interactive pages)
- **Tier 5** — deep Streamlit patterns requiring full UX redesign (auth flow, multi-page nav, app.py)

Builder applies rule mechanically, then adjusts ±1 where module comments indicate complexity.

## Patterns Worth Preserving

### Offline-First Sync (data_layer.py + sync_engine.py)

Google Sheets is the authority; SQLite is the local cache. Bidirectional sync with conflict
resolution. Pages work even when Sheets is slow/down. This pattern is valuable for LepiOS —
adapt as Supabase-primary + local cache.

### Circuit Breaker (utils/circuit_breaker.py, 232 lines)

Per-service state machine (CLOSED → OPEN → HALF_OPEN). Already ported to LepiOS as
`lib/ollama/circuit.ts`. The Streamlit version is the reference; extend to cover Sheets,
Keepa, SP-API in LepiOS.

### Action Dispatch System (utils/actions.py, 1,098 lines)

Centralized action registry with safety levels (open/confirm/approve). Clean abstraction
over "trigger this operation". LepiOS equivalent: task_queue + coordinator invocation.

### Health Check Per Section (utils/health_check.py, 327 lines)

Per-section status (green/yellow/red) cached 24h. Prevents hammering APIs on every render.
LepiOS should port as a status_card query pattern.

### Retry Conventions (manual, no tenacity)

3 attempts, exponential backoff, in: amazon.py, keepa_api.py, builder_bot.py.
No external retry library. Pattern: `for attempt in range(3): try: ... except: sleep(2**attempt)`.
LepiOS port: use the same manual pattern or adopt a lightweight TypeScript equivalent.

### Error Isolation (debug.py + try/except everywhere)

~200 files: pages never crash, all errors caught, logged, displayed.
Every page returns empty/None on failure with a user-visible warning, not a traceback.
This is the right pattern for LepiOS API routes too — always return 200 with error details
in the JSON body rather than 500.

## Chunking Strategy

### .py files — function-level

Split on `^def ` and `^    def ` (top-level functions + class methods). Each chunk:

```
# File: utils/amazon.py — function: get_orders
def get_orders(self, ...):
    ...
```

Minimum chunk size: 5 lines (skip trivial getters). Maximum: 150 lines (split long functions
at blank-line boundaries). A 3,373-line file like PageProfit.py produces ~40-60 chunks.

Estimated total chunks: ~3,000-5,000 rows in knowledge table with domain='streamlit_source'.

### .md files — file-level

Embed each .md as a single chunk. Max 6,000 chars; truncate at section boundary if longer.
