## What Works (the 80%)

### Atomic claim — `FOR UPDATE SKIP LOCKED`

`supabase/migrations/0016_add_pickup_fns.sql:29` — the `claim_next_task()` RPC uses
`FOR UPDATE SKIP LOCKED` in its subquery. Two concurrent pickup invocations cannot claim the
same task. The second run gets zero rows and exits cleanly. No application-level race window.

### Priority + FIFO ordering

`0016_add_pickup_fns.sql:27-28` — `ORDER BY priority ASC, created_at ASC`. Priority 1 is
always claimed before priority 5 or 10. Ties broken by creation time (FIFO). Index at
`0015_add_task_queue.sql:69-70` supports this query efficiently.

### Stale recovery (10-minute timeout)

`lib/harness/pickup-runner.ts:92` calls `reclaimStale()` at the top of every run.
`0016_add_pickup_fns.sql:64` — `COALESCE(last_heartbeat_at, claimed_at) < NOW() - INTERVAL '10 minutes'`
reclaims tasks that went silent. Retry counter incremented per reclaim; tasks cancelled at
`max_retries` (default 2) with `error_message='stale claim: max retries exhausted'`.

### Attribution wired (just shipped)

`lib/harness/pickup-runner.ts:135-144` — `void recordAttribution(...)` fires after every
successful claim. Attribution table live as of migration 0020 (2026-04-24).

### Feature flags

`app/api/cron/task-pickup/route.ts` — `TASK_PICKUP_ENABLED`, `TASK_PICKUP_DRY_RUN`,
`HARNESS_REMOTE_INVOCATION_ENABLED` gates. Dry run prefixes Telegram messages. Clean.

### basic `agent_events` logging

`lib/harness/pickup-runner.ts:46-74` — logs every run outcome: empty queue, validation fail,
task claimed, Telegram notify fail. `domain='orchestrator'`, `action='task_pickup'`.

## What's Missing (the 20%)

### Gap 1 — Cron fires once per day (CRITICAL — estimated 12–15% of the missing 20%)

**Evidence:** `vercel.json:20-21` — `"schedule": "0 16 * * *"` = 16:00 UTC daily.

**Production symptom:** Task `9038e0f7` queued 2026-04-23 17:18 UTC, claimed 2026-04-24
16:02 UTC. **Wait time: 81,872 seconds = 22.7 hours.** A "task pickup" system that batches
once per day is a batch job, not an autonomous harness.

All 7 routes in `vercel.json` use daily schedules. This is consistent with Vercel Hobby plan
(daily-only cron frequency) OR a deliberate choice on Pro plan (which supports hourly).
**This is the pending_colin_q — see Phase 1c below.**

**20% Better target:** Max pickup latency of 1 hour for priority 1-2 tasks; ideally under
30 minutes for the harness to feel autonomous.

---

### Gap 2 — Heartbeat written only at task completion, not periodically (CRITICAL — ~5%)

**Evidence:**

- Production `task_queue` rows — every completed task shows `last_heartbeat_at = completed_at`
  (e.g., task `9038e0f7`: `claimed_at=16:02`, `last_heartbeat_at=16:13`, `completed_at=16:13`).
  The coordinator writes `last_heartbeat_at` exactly once at the end of its run.
- `lib/harness/task-pickup.ts:53-60` — `heartbeat(taskId)` function is defined but the
  coordinator is only calling it at completion, not during execution.
- **No heartbeat-in-progress evidence:** Every running task in production either has
  `last_heartbeat_at=null` (still running) or `last_heartbeat_at=completed_at` (finished).

**Production casualty:** Task `47a77122` (version.ts test) — claimed Apr 22 15:17 UTC,
`last_heartbeat_at=null` throughout, stale-reclaimed at Apr 22 16:06 (49 minutes later → the
daily cron ran again before coordinator finished). Re-stale at Apr 23 13:27.
Final state: `status='cancelled'`, `retry_count=2`, `max_retries=2`,
`error_message='stale claim: max retries exhausted'`. The coordinator ran but the task was
pulled out from under it twice.

**Why this happens:** A coordinator session can take 30-60+ minutes (multiple phases,
acceptance doc writing, builder invocation). The 10-minute stale window expires during normal
execution. If the coordinator calls `heartbeat()` every 5 minutes, the stale window never
triggers during active work. If it only calls at completion, the window will false-alarm on
any session >10 minutes.

**Fix:** Coordinator must call `heartbeat(taskId)` at least every 5 minutes during execution —
specifically after each Phase transition (1a, 1b, 1c, 1d, builder kick-off, builder poll).
This is a coordinator.md change, not a schema change.

### Gap 3 — No pickup latency or queue depth in `agent_events` (MEDIUM — ~2-3%)

**Evidence:** Production `agent_events` meta field for every `task_pickup` event:

```json
{ "run_id": "72d505a7-...", "claimed_task_id": "9038e0f7-..." }
```

That's it. No latency, no queue depth, no stale reclaim count.

**What's missing:**

- `latency_ms`: `claimed_at - created_at` for the claimed task — how long did this task wait?
- `queue_depth`: how many `status='queued'` rows existed before the claim — is the queue
  building up?
- `stale_reclaimed`: count of rows reclaimed by `reclaimStale()` this run
- `cancelled_this_run`: count of tasks cancelled for exceeding max_retries

Without these, morning_digest can't say "pickup latency p99 was 22h yesterday" or "3 tasks
stuck in queue for >2 days." The F18 benchmark can't be verified.

---

### Gap 4 — Stale recovery not logged as a distinct event (MINOR — ~1%)

**Evidence:** Querying `agent_events` for stale-recovery events returns 0 rows. The reclaim
function (`reclaimStale()`) runs on every pickup invocation but logs nothing separately.
If it reclaimed 3 tasks, you'd never know without querying `task_queue` directly.

**Fix:** When `reclaimStale()` returns non-zero rows, log one `agent_events` row with
`action='task_pickup_stale_reclaimed'`, `meta.count=N`, `meta.task_ids=[...]`.

## Phase 1b — Digital Twin Q&A

All ambiguities pattern-match from existing principles except one:

**Answered by twin (no Colin escalation needed):**

- Q: Should `max_retries` change? → No. 2 retries matches CLAUDE.md global retry limit. Stale
  recovery IS retry — when coordinator heartbeats correctly, no retries will be needed at all.
- Q: Should failed tasks auto-retry? → No. `status='failed'` means the coordinator explicitly
  gave up, not just went silent. Auto-retrying explicit failures causes loops. Alert Colin instead.
- Q: Increase stale window from 10 to 15 minutes? → Yes, as a safety buffer for heartbeat
  jitter. 15 minutes = 3× the intended 5-min heartbeat interval. Still fast recovery for truly
  dead sessions.
- Q: Should `heartbeat()` be added to coordinator.md phases or added as a wrapper? → Add to
  coordinator.md as a mandatory action at each phase transition. No new wrapper needed — the
  function already exists at `lib/harness/task-pickup.ts:53`.

(`0 * * * *`)? Vercel Pro supports hourly; Hobby only supports daily. Which plan is active?

(`/pickup` → Colin taps it when he wants to kick off a queued task) rather than scheduling it.

The improvement-engine acceptance doc referenced `notifications-drain` running "every minute"
but it's not in `vercel.json`. Is `notifications-drain` triggered by something other than a
Vercel cron (e.g., manually, by `after()`, or removed)? This affects whether we should extend
drain to trigger pickup rather than changing the cron schedule.
