## Phase 1a-i — F17 and F18 Compliance Requirements

### F17 — Behavioral Ingestion Rule

Every module must justify its contribution to the path probability engine. Before building, ask
"what does this feed the path engine?" If the answer is nothing, reconsider.

**This module's F17 answer:**  
The 20% Better Loop Engine feeds the path engine in three concrete ways:

1. **Coordinator/builder blind spot signal** — every proposal identifies a category where the
   autonomous loop failed. These are labeled datapoints about the current accuracy boundaries of
   the harness. Over time, a declining proposal volume in a category = the loop has self-corrected
   in that domain. Rising volume = degradation signal.

2. **Approval/rejection training signal** — Colin's thumbs on proposals are high-quality labels:
   "this was worth fixing" vs. "not a real problem." These train future quality bar inference
   without requiring Colin to articulate rules.

3. **Corpus gap pipeline** — proposals in category `twin_corpus_gap` feed directly back into
   the Twin ingestion backlog. The engine identifies its own knowledge gaps automatically.

F17 compliance is native to this module, not retrofitted.

### F18 — Measurement Framework Rule

Every module declares: a metric, a benchmark before writing code, a capture method, a surfacing
path, and an alert threshold.

**Required before any code is written:**
- Primary metric and benchmark must be defined
- Capture method wired on day one to `agent_events`
- Surfacing path defined (not "I could write a query")
- Alert threshold set

These constraints are carried forward verbatim into the acceptance doc. F18 cannot be treated as
a post-hoc annotation.

## Phase 1c — 20% Better Feedback Loop

The 20% Better loop is non-optional for new harness components that have no Streamlit predecessor
(greenfield). For greenfield, the question becomes: "How do I make this 20% better than the
obvious naive implementation?"

### Correctness

Naive implementation: analyzer reads task_queue metadata, proposes based on keywords.  
**20% better:** Analyzer reads the _actual_ acceptance doc file (path stored in task_queue meta)
and the actual agent_events for the task_id, not just metadata fields. This catches failures that
don't appear in metadata (e.g., grounding mismatches logged in sprint-state, not agent_events).

Naive implementation: proposals based on last chunk only.  
**20% better:** Deduplicator scans _all open proposals_ before inserting, not just same-chunk.
Same pattern recurring across 3 chunks = a "chronic" flag that escalates severity automatically.

### Performance

Naive implementation: analyzer runs synchronously on chunk completion, blocking the task pickup loop.  
**20% better:** Analyzer runs as a fire-and-forget cron side-effect, not inline with task completion.
Trigger is the status change event, but analysis happens in the next cron cycle. This prevents the
analyzer's latency from becoming a bottleneck in the main task loop.

### UX

Naive implementation: one Telegram message per proposal (spam risk).  
**20% better:** One Telegram message per chunk, regardless of proposal count. Inline keyboard:
`[Approve all] [Review each] [Dismiss]`. Colin sees "3 proposals for Chunk D" not "proposal 1 of 3,
proposal 2 of 3, proposal 3 of 3" in separate messages.

Naive implementation: proposal text is raw JSON.  
**20% better:** Plain-language summary per proposal with the concrete action stated in one sentence.
Colin reads on mobile; dense JSON is useless.

### Extensibility

Naive implementation: category list hardcoded in the proposer.  
**20% better:** Category list in a centralized constants module with `// TODO: tune with real data`
per Principle 11. Adding a new category is a one-line change.

Naive implementation: severity hardcoded as "meaningful" for all proposals.  
**20% better:** Severity derived from recurrence count — 1st occurrence = `nice_to_have`, 2nd
occurrence of same category = `meaningful`, 3rd+ = `blocking`. This auto-escalation is the
"chronic" signal.

### Data model

Naive implementation: proposals stored in a new table.  
**20% better:** Proposals stored in the existing `task_queue` table as `task_type='improvement_proposal'`.
This means coordinator's existing pickup loop can see them; no new infrastructure needed. Check-Before-Build confirms `task_queue` already exists and is the right abstraction.

### Observability

Naive implementation: proposals land in task_queue, no visibility until coordinator processes them.  
**20% better:** Three surfacing paths:
1. Morning digest: "improvement loop: N proposals queued, M pending Colin review"
2. `/autonomous` page tile: proposal stats (total queued, approval rate, avg time-to-implementation)
3. Telegram notify on chunk completion (per Notifier component above)
