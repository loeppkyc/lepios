## Scope

Three gap closures to move task pickup from 80% to 100%:

1. **Cron frequency** — change from daily (`0 16 * * *`) to hourly (`0 * * * *`), reducing
   max pickup latency from 24 hours to 1 hour. _Blocked on Q1 — see Open Questions._
2. **Coordinator heartbeat** — add mandatory `heartbeat(taskId)` calls to `coordinator.md`
   at each phase transition so active coordinator sessions never trip the 10-minute stale
   window. Zero schema changes required.
3. **Observability** — add `latency_ms`, `queue_depth`, `stale_reclaimed`, `cancelled_this_run`
   to the `agent_events` meta on every pickup run so morning digest can surface queue health.

**One acceptance criterion:** After this ships:

- A task queued at any hour is claimed within 65 minutes (one cron cycle + 5-minute buffer).
- A coordinator session running for 45 minutes does NOT get its task stale-reclaimed.
- The `agent_events` row for every pickup contains `latency_ms` and `queue_depth`.

## Gap 1 — Cron Frequency

**File:** `vercel.json`

**Change:**

```diff
-    { "path": "/api/cron/task-pickup", "schedule": "0 16 * * *" }
+    { "path": "/api/cron/task-pickup", "schedule": "0 * * * *" }
```

`0 * * * *` = at the top of every hour, 24 times per day.

**Why hourly (not `*/30` or `*/5`):**

- Vercel Pro maximum sub-daily frequency = hourly. `*/30` or `*/5` requires Enterprise.
- Hourly is a 24× improvement over daily (24h max → 1h max). Good enough for a personal OS.
- One pickup per hour = 24 database reads per day. Negligible load.

**Acceptance:** After deploy, a task queued at any minute of an hour is claimed by the next
hour's pickup run. Verify by queueing a test task and confirming `claimed_at - created_at < 65`
minutes in the next pickup event.

## Gap 2 — Coordinator Periodic Heartbeat

**File:** `.claude/agents/coordinator.md`

**What to add:** Immediately after the `# What you do (the loop)` section that describes
the coordinator phases, add a **Heartbeat Rule** (non-negotiable, same tier as Phase 0):

```markdown
## Heartbeat Rule (Tier 0 — required for every claimed task)

When you begin work on a claimed task (i.e., a task_queue row with status='claimed'
and a task_id you are operating on), you MUST call the heartbeat endpoint after every
major phase transition to prevent the task from being stale-reclaimed.

Call pattern (Bash tool):
curl -s -X POST https://lepios-one.vercel.app/api/harness/task-heartbeat \
 -H "Authorization: Bearer $CRON_SECRET" \
 -H "Content-Type: application/json" \
 -d '{"task_id": "<task_id>"}'

Mandatory call points:

1. After Phase 1a completes (study doc written)
2. After Phase 1b completes (twin Q&A done)
3. After Phase 1c completes (acceptance doc ready, before escalating to Colin)
4. After builder is invoked (fire-and-forget; immediately after fireCoordinator returns)
5. If any phase takes >5 minutes (set a mental checkpoint: after every file read batch)

If $CRON_SECRET is not in environment: skip the curl and log an agent_events row:
action='heartbeat_skipped', status='warning', meta.reason='missing_cron_secret', meta.task_id=<task_id>
Do NOT fail or abort — continue working. Missing CRON_SECRET is a non-fatal environment gap.

Why this matters: the stale recovery window is 15 minutes. A phase that takes >15 minutes
without a heartbeat will cause the pickup cron to re-claim your task and increment retry_count.
At retry_count=2 the task is cancelled. Heartbeating prevents this entirely.
```

**New API route needed:** `app/api/harness/task-heartbeat/route.ts` (build new — lightweight)

```typescript
// POST /api/harness/task-heartbeat
// Body: { task_id: string }
// Auth: verifyHarnessSecret (same pattern as other harness routes)
// Action: UPDATE task_queue SET last_heartbeat_at = now() WHERE id = task_id AND status = 'claimed'
// Returns: { ok: true } or { ok: false, error: string }
```

This is a thin wrapper around the existing `heartbeat()` function in `lib/harness/task-pickup.ts`.
Builder must read that function before writing the route — do NOT reimplement the logic.

**Why a new route (not calling `heartbeat()` inline in coordinator.md):** The coordinator
runs as an Anthropic Routines session and calls Vercel API routes over HTTP (same pattern as
all other coordinator→harness communication). It cannot import TypeScript modules directly.
The `heartbeat()` function already exists — the route just exposes it via HTTP.

**Stale window: extend from 10 → 15 minutes**

```sql

RETURNS TABLE(task_id UUID, action TEXT)
LANGUAGE plpgsql
AS $$
-- ... (builder reads the existing function from 0016_add_pickup_fns.sql and replaces only the interval)
-- Change: INTERVAL '10 minutes' → INTERVAL '15 minutes'
```

**Why 15 minutes:** 3× the 5-minute heartbeat cadence. A coordinator that heartbeats every
5 minutes will never approach 15 minutes of silence. But if the routine environment is slow
or a phase runs long, the extra 5 minutes prevents false positives.

Builder must read `0016_add_pickup_fns.sql` in full before writing this migration.
The `reclaim_stale_tasks()` function must be recreated in full (Postgres `CREATE OR REPLACE`
requires the full function body). Do NOT write a partial replacement.

**Acceptance:**

- Hit `POST /api/harness/task-heartbeat` with a valid task_id. `task_queue.last_heartbeat_at`
  updates within 1 second. Returns `{ ok: true }`.
- With a fake task_id: returns `{ ok: false, error: 'task not found or not claimed' }` (no 500).
- With no auth token: returns 401.
- Query confirms stale window is now 15 minutes:
  ```sql
  SELECT prosrc FROM pg_proc WHERE proname = 'reclaim_stale_tasks';
  -- Confirm: contains '15 minutes', not '10 minutes'
  ```

## Gap 3 — Observability Metrics

**What to add to the `logEvent` call on successful claim:**

1. `latency_ms` — time from task's `created_at` to `claimed_at`:

   ```typescript
   const latencyMs = task.created_at ? Date.now() - new Date(task.created_at).getTime() : null
   ```

2. `queue_depth` — count of remaining `status='queued'` rows AFTER the claim:

   ```typescript
   const { count: queueDepth } = await db
     .from('task_queue')
     .select('id', { count: 'exact', head: true })
     .eq('status', 'queued')
   ```

   Run this AFTER the claim so the claimed task is not counted.

3. `stale_reclaimed` — the count of rows returned by `reclaimStale()` (currently returned
   but not logged). Pass this into the `logEvent` call.

**Updated `logEvent` meta shape:**

```typescript

  run_id: runId,
  claimed_task_id: taskId,
  latency_ms: latencyMs,           // NEW

```

**Also add: stale reclaim event when count > 0:**

## Measurement (F18)

```
module: task-pickup
metric_name: Pickup latency p95 + false stale rate
units:
  - latency_ms: ms from task.created_at to agent_events.occurred_at (≈ claimed_at)
  - false_stale_rate: stale reclaims on tasks that were NOT abandoned / total claims
capture_method: |
  Latency: agent_events WHERE action='task_pickup' → meta.latency_ms
  False stale: agent_events WHERE action='task_pickup_stale_reclaimed'
    AND meta.reclaimed is non-empty; correlate reclaimed task_ids with tasks
    that eventually completed (not cancelled) — those are false stales
benchmark:
  - latency p95 ≤ 3,600,000 ms (1 hour) after moving to hourly cron
    [baseline: 22.7h / 81,872s from production data 2026-04-24]
  - false stale rate = 0 after periodic heartbeat is live
    [baseline: 1 false stale in last 10 tasks = 10%]
surfacing_path: |
  morning_digest: "pickup: last 24h latency avg Xmin, queue depth N, false stales: 0"
  Query: SELECT AVG(meta->>'latency_ms'::numeric), MAX(meta->>'latency_ms'::numeric),
         AVG(meta->>'queue_depth'::numeric)
    FROM agent_events
    WHERE action = 'task_pickup' AND occurred_at > now() - interval '24 hours'
alert_threshold: |
  - Any task with latency_ms > 7,200,000 (2 hours) → queue is stuck; Telegram alert
  - false_stale_rate > 0 for 3 consecutive days → heartbeat not being called; alert
```

## Engine-Feeding (F17)

| Signal                                   | Path to engine                                                                                                                      |
| ---------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `latency_ms` per task                    | Time-to-pickup as a state signal — slow pickup = Colin is offline or coordinator is blocked                                         |
| `queue_depth` per run                    | Queue depth growth rate = signal for coordinator throughput. If depth grows without decreasing, coordinator is not completing tasks |
| `false_stale_rate`                       | Coordinator health signal — if non-zero, coordinator is not heartbeating (execution environment issue)                              |
| `task.source` distribution at claim time | Which sources (manual, improvement_engine, coordinator) generate tasks most — path probability input                                |
| Cancellations (max retries exhausted)    | Task abandonment rate per source — training signal for "which kinds of tasks fail silently"                                         |
