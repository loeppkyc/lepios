## What the Streamlit Implementation Does

### Auth flow (`get_gmail_service()`)

Two-path auth:

- **Cloud path**: reads `st.secrets["gmail"]` → builds `google.oauth2.credentials.Credentials`
  with `refresh_token` → calls `creds.refresh(Request())` → returns `build("gmail", "v1", ...)`
- **Local dev path**: reads `gmail_token.json` from `.streamlit/` first (written by `gmail_oauth_setup.py`),
  falls back to secrets.toml if file missing or token expired

Scopes: `gmail.readonly` only. No write access.

Required secrets (3 fields):

- `client_id` — OAuth2 app client ID
- `client_secret` — OAuth2 app client secret
- `refresh_token` — long-lived refresh token (never expires unless revoked)
- `token_json` — optional full token blob for Cloud; individual fields are the fallback

**LepiOS mapping:** `client_id` → `GOOGLE_CLIENT_ID`, `client_secret` → `GOOGLE_CLIENT_SECRET`,
`refresh_token` → `GOOGLE_REFRESH_TOKEN`. No `token_json` needed — refresh_token flow only.

### Scan 1 — Invoice/attachment scan (`scan_invoices()`)

**Two-pass Gmail query approach:**

Pass 1 — keyword query:

```
after:{date} has:attachment subject:(invoice OR receipt OR bill OR folio OR statement OR confirmation OR order)
```

Pass 2 — known-sender query (catches invoices without keywords in subject):

```
after:{date} has:attachment (from:marriott.com OR from:telus.com OR from:anthropic.com OR ...)
```

Both passes are unioned; dedup by `{message_id}|{attachment_name}`.

**Attachment filtering** (noise reduction, important for LepiOS port):

- Skip filenames starting with `outlook-`, `image0`, `logo` (email signature logos)
- Skip `screenshot*`, `whatsapp*`, `gmail_-_*` (forwarded screenshots)
- Skip `image*.png/jpg/jpeg` (generic signature images)
- Skip images < 5KB by `body.size` (inline signature icons)
- KEEP only `.pdf`, `.jpg`, `.jpeg`, `.png`

**Return shape:** `{message_id, date, from, subject, attachment_name}`

**KNOWN_INVOICE_SENDERS** (20 domains):
`marriott.com`, `telus.com`, `polarhq.com`, `amazon.ca`, `lego.com`, `bestbuy.ca`,
`staples.ca`, `canadapost-postescanada.ca`, `adobe.com`, `anthropic.com`,
`microsoft.com`, `dropbox.com`, `google.com`, `apple.com`, `shopify.com`,
`keepa.com`, `sellerboard.com`

### Scan 2 — Inline receipt scan (`scan_inline_receipts()`)

For stores that put the receipt in the email body (no meaningful attachment).

**Query approach:**
Pass 1 — known retail senders + receipt-like subjects (negative: offer/promo/coupon/newsletter)
Pass 2 — broad `subject:receipt` from any sender (catches stores not in KNOWN_INLINE_SENDERS)

Note: no `has:attachment` filter on Pass 1 — some stores (e.g. London Drugs) embed a logo
as an attachment but the actual receipt is in the body.

**Body extraction (`_extract_body_text()`):**

- Prefers `text/plain` MIME part
- Falls back to `text/html` with strip-tags + collapse-whitespace conversion
- Handles nested `multipart/` recursively
- Caps output at 4,000 chars (sized for Claude context injection)

**Noise filter:** skip if `len(body.strip()) < 100` (marketing blurb, not a receipt)

**Return shape:** `{message_id, date, from, subject, body_text}`

**KNOWN_INLINE_SENDERS** (40 domains):
All major Canadian retailers — London Drugs, Walmart, Costco, Canadian Tire, Best Buy,
Staples, Home Depot, Sport Chek, Bay, Winners/Marshalls, Dollarama, major grocers,
pharmacies (Shoppers, Pharmasave, Rexall), Chapters/Indigo, Amazon, eBay, Etsy,
Lego, Toys R Us, Cineplex, telecom (Telus, Shaw, Rogers, Bell), delivery (SkipTheDishes,
Uber Eats, DoorDash), travel (Airbnb, Expedia, Booking), parking (Impark).

### Trust level system (`load_sender_trust_levels()`, `save_learned_sender()`)

Google Sheets tab: `📧 Known Senders`
Columns: `Domain | Vendor | First Seen | Trust Level`
Trust levels: `trusted` (recurring expense), `skip` (never an expense), `review` (default)

On first encounter of a new sender: calls `save_learned_sender(domain, vendor, "review")`
which appends to the sheet. On subsequent scans: `load_learned_senders()` reads col A and
merges those domains into both scan queries (cached 5 min).

Legacy rows without col D default to `"trusted"` (backward compat).

---

### Attachment download (`download_attachment()`)

`service.users().messages().attachments().get()` → base64url decode → bytes.
Recursive MIME part walker to find attachment by filename. Silently returns `None` on error.

## Supabase Table Needed: `gmail_known_senders`

Replaces the Google Sheets "📧 Known Senders" tab.

```sql
CREATE TABLE gmail_known_senders (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  domain      text NOT NULL UNIQUE,
  vendor      text NOT NULL,
  trust_level text NOT NULL DEFAULT 'review' CHECK (trust_level IN ('trusted', 'skip', 'review')),
  first_seen  date NOT NULL DEFAULT CURRENT_DATE,
  updated_at  timestamptz NOT NULL DEFAULT now()
);
```

`load_learned_senders()` → `SELECT domain FROM gmail_known_senders WHERE trust_level != 'skip'`
`save_learned_sender()` → `INSERT ... ON CONFLICT (domain) DO UPDATE SET trust_level = ...`
`load_sender_trust_levels()` → `SELECT domain, vendor, trust_level FROM gmail_known_senders`

## F17 Signal (Behavioral Ingestion)

Gmail scanner feeds two engine signals:

1. **Expense velocity**: frequency and amount of receipt/invoice emails → proxy for spending
   rate and category mix (Amazon vs. groceries vs. SaaS tools)
2. **Vendor presence**: which vendors send invoices → business activity indicator (e.g.
   Anthropic invoice = AI tooling spend, Shopify invoice = active store)

Proposed `agent_events` action: `gmail.scan_invoices` with `meta: {found, after_date, scan_type: 'invoice'|'inline'}`

## Gaps Not in Streamlit (20% Better Opportunities)

1. **Structured amount extraction**: Streamlit returns `body_text` raw — no amount parsing.
   LepiOS could run a lightweight regex or Claude extraction pass to pull dollar amounts,
   converting inline receipts into structured expense candidates.
2. **Dedup across scans**: Streamlit re-fetches all messages from `after_date` every scan
   with no memory of what's already been imported. LepiOS can store `message_id` in Supabase
   and skip already-imported messages on subsequent scans.
3. **Attachment storage**: Streamlit downloads attachments on demand only. LepiOS can
   immediately store PDFs to Supabase Storage after scan, making them available for
   downstream OCR without re-fetching from Gmail.
4. **Telegram notification on new invoices**: Streamlit has no alerting. LepiOS can push
   a Telegram digest when a scan finds new invoices, listing vendors + attachment filenames.
5. **Scan scheduling**: Streamlit scans are manual (user-triggered). LepiOS can run as a
   daily cron (morning, after overnight Gmail activity) and store results for the dashboard.

## Recommended LepiOS Architecture

```
app/api/gmail/scan/route.ts        — POST: runs both scan passes, returns results
app/api/gmail/download/route.ts    — POST: downloads attachment bytes by message_id + filename
lib/gmail/client.ts                — getGmailService(), buildCredentials()
lib/gmail/scan.ts                  — scanInvoices(), scanInlineReceipts(), extractBodyText()
lib/gmail/senders.ts               — KNOWN_INVOICE_SENDERS, KNOWN_INLINE_SENDERS constants
supabase/migrations/0022_add_gmail_known_senders.sql
```

Auth: server-side only (API routes). No client-side Gmail calls.
Cron: daily at 08:00 UTC via `vercel.json` → `/api/cron/gmail-scan`

## Pending Questions for Phase 1b

**Q1 (scope):** Does LepiOS need attachment _download + storage_ in v1, or just the
scan manifest (message_id, date, from, subject, filename)? The Streamlit app downloads
on demand; LepiOS could store to Supabase Storage immediately. Adds storage cost but
makes PDFs available for OCR without re-auth.

**Q2 (trust store):** Migrate existing trust levels from Google Sheets `📧 Known Senders`
to `gmail_known_senders` Supabase table? Or start fresh and let LepiOS learn from v1 scans?
Streamlit has an unknown number of existing rows.

**Q3 (scan trigger):** Manual scan only (API route Colin calls), or also daily cron?
Daily cron means the dashboard always has fresh data; manual-only is simpler for v1.

**Q4 (amount extraction):** Include structured amount extraction (regex + optional Claude
pass) in v1, or raw `body_text` only (Streamlit parity first)?
