## Scope

One sentence: Build an engine that fires after every chunk completion, analyzes the chunk's
audit trail, generates 0–N structured improvement proposals, deduplicates against open proposals,
queues them in `task_queue`, and notifies Colin via Telegram (one message per chunk, inline
keyboard for bulk approve/dismiss).

**One acceptance criterion:** After a chunk status changes to `completed` or `grounded`, the
engine fires within 60 seconds, produces a structured JSON audit object, queues any non-duplicate
proposals into `task_queue` with `task_type='improvement_proposal'`, and sends Colin one Telegram
message via `outbound_notifications` if any proposals were queued. Running the engine twice on
the same chunk produces no duplicate task_queue rows.

---

## Out of Scope

- Automated proposal implementation (coordinator still picks up proposals; no auto-apply)
- Retrospective scoring / proposal effectiveness tracking in v1 (see Measurement section —
  tracked but not surfaced beyond morning digest in v1)
- Fine-tuning or corpus updates from approvals (downstream of this module)
- `/autonomous` page tile (proposal stats tile deferred — add after first 5 proposals cycle
  through approval)
- Proposal review UI beyond Telegram inline keyboard
- Any write to a source-of-truth table (ledger, audit, tax) — proposals are metadata only

**Why deferred:** Proposal UI, corpus-update loop, and effectiveness scoring all require
observing proposal outcomes first. v1 establishes the pipeline; v2 closes the feedback loop.

## Component 1 — Trigger

**What it does:** Detects when a task_queue row transitions to `status='completed'` or
`status='grounded'` and fires the Analyzer.

**Implementation options (builder must pick one and document the decision):**
- Option A: Extend the existing notifications-drain cron (`app/api/harness/notifications-drain`)
  to also check for recently-completed task_queue rows and invoke the Analyzer inline.
- Option B: Separate cron route `app/api/harness/improvement-engine` running on the same
  every-minute schedule (Vercel Hobby plan allows multiple crons).
- Option C: Trigger from the task-pickup route at the moment a task is marked complete.

**Builder constraint:** Whatever option is chosen, the trigger must use `claimed_by` and
`completed_at` fields already present in task_queue. Do not add new columns to task_queue
without flagging it as a migration (escalate).

**Trigger cadence (Q4 — resolved 2026-04-24):** Per-chunk within 60 seconds of completion.
A separate nightly digest cron is allowed as an additive summary of the day's proposals +
approval/rejection rates, but is NOT the analyzer trigger — it does not replace per-chunk
firing. Nightly digest is out of scope for v1.

**Acceptance:** Engine fires within 60 seconds of a `completed_at` timestamp being written to
task_queue. Measured via `agent_events` entry logged by the engine with
`action='improvement_engine.triggered'`, `status='success'`, `meta.task_id=<uuid>`,
`meta.lag_ms=<elapsed since completed_at>`.

## Component 2 — Analyzer

**What it does:** Given a completed chunk's task_queue row, reads its full audit trail and
produces a structured JSON audit object.

**Inputs the Analyzer reads:**
1. `task_queue` row — `metadata` field for `chunk_id`, `sprint_id`, `acceptance_doc_path`
2. Acceptance doc file at `acceptance_doc_path` — full text read (path must exist; if not,
   Analyzer logs `action='improvement_engine.audit_incomplete'` and exits without proposing)
3. `agent_events` WHERE `meta->>'task_id' = <task_id>` — all events for this task
4. `agent_events` for the same sprint/chunk scope — escalation events, notification failures,
   twin escalations, ollama failures
5. `docs/sprint-state.md` — grounding status for the chunk

**Audit object shape:**

```typescript
interface ChunkAudit {
  chunk_id: string
  sprint_id: string
  task_queue_id: string
  acceptance_doc_found: boolean
  grounding_status: 'passed' | 'passed_with_limitation' | 'failed' | 'not_yet'
  grounding_mismatches: number  // count of Colin corrections at grounding time
  escalations_to_colin: number  // coordinator escalations logged in agent_events
  twin_escalations: number      // twin.ask → escalate:true events for this chunk
  notification_failures: number // notification_failed events
  ollama_failures: number       // ollama.generate/embed failures
  review_bypasses: number       // pre-commit bypasses (from review-skips.md parsing, optional)
  spec_corrections: number      // grounding corrections that trace to spec omissions
  analyzed_at: string           // ISO timestamp
}
```

**Acceptance:** For every completed chunk processed, the Analyzer produces a `ChunkAudit`
object and logs it to `agent_events` with `action='improvement_engine.audit_complete'`,
`status='success'`, `output_summary=JSON.stringify(audit)`.

## Component 3 — Proposer

**What it does:** Reads the `ChunkAudit` and generates 0–N `ImprovementProposal` objects.

**Proposal shape:**

```typescript
interface ImprovementProposal {
  category: 'process' | 'code_pattern' | 'test_coverage' | 'doc_gap' | 'tooling' | 'twin_corpus_gap' | 'security' | 'reliability'
  severity: 'blocking' | 'meaningful' | 'nice_to_have'
  concrete_action: string  // must name (a) specific file/doc/process AND (b) exact change — both required
  engine_signal: string    // what this teaches the path probability engine (F17)
  measurement: string      // metric + before value + target (F18)
  source_chunk_id: string  // back-reference
  fingerprint: string      // category + first 60 chars of concrete_action (for deduplication)
}
```

**Severity derivation:**
- First occurrence of a category+fingerprint pattern: `nice_to_have`
- Second occurrence (found by Deduplicator — recurrence_count = 1): `meaningful`
- Third+ occurrence (recurrence_count ≥ 2): `blocking`

This auto-escalation means the engine flags chronic failures without a human having to notice
the pattern manually.

**Proposal generation rules:**

| Audit signal | Category | Concrete action template |
|---|---|---|
| `grounding_mismatches > 0` AND mismatch traces to spec field | `doc_gap` | "Add numeric field definition table to acceptance doc for [chunk_type] chunks specifying OrderStatus filter, pending handling, and SC match target" |
| `grounding_mismatches > 0` AND mismatch traces to timezone | `doc_gap` | "Acceptance docs for date-comparison chunks must specify timezone, boundary test cases, and exact date range purpose" |
| `twin_escalations > 2` | `twin_corpus_gap` | "Ingest [topic] decisions into Twin corpus — [N] escalations in chunk [id] were corpus gaps, not personal decisions" |
| `notification_failures > 0` | `tooling` | "Wire coordinator to detect sandbox environment at startup and log notification_skipped instead of failing" |
| `ollama_failures > 3` | `tooling` | "Add Ollama liveness check to morning_digest — if Ollama unreachable, alert Colin before overnight runs begin" |
| `review_bypasses > 0` AND reason = ANTHROPIC_API_KEY | `tooling` | "Export ANTHROPIC_API_KEY from .env.local into pre-commit hook env — add sourcing line to .husky/pre-commit" |
| `escalations_to_colin > 3` AND chunk is port (not greenfield) | `process` | "Run Phase 1a Streamlit study before writing acceptance doc for [chunk_type] chunks — coordinator.md Phase 1a is mandatory but was skipped" |
| Any auth/secret handling change without RLS review | `security` | "Add RLS policy review to acceptance checklist for chunks touching [table] — auth change in [file] shipped without safety agent sign-off" |
| `ollama_failures > 3` OR harness silent-degradation | `reliability` | "Add circuit-breaker to [component] — [N] failures in last 14 days caused silent degradation with no alert" |

**Acceptance:** Proposer generates ≥1 proposal for any chunk where `grounding_mismatches > 0`
OR `escalations_to_colin > 3` OR `twin_escalations > 2` (i.e., any chunk with a measurable
failure pattern). Proposal count = 0 for clean chunks (no mismatches, no excessive escalations)
is correct behavior — do not force proposals.

**Concrete action bar (Q2 — resolved 2026-04-24):** A `concrete_action` string must satisfy
BOTH criteria or the engine self-rejects it before queueing:

- (a) Names the specific file, doc section, or process step
- (b) Names the exact change to make to it

"Improve spec quality" → rejected (fails a and b).
"Update `coordinator.md §8.4` to require timezone defined for any date-comparison chunk" →
accepted (names the file+section AND the exact change).
The engine validates this before INSERT. Rejected proposals are logged to `agent_events` with
`action='improvement_engine.proposal_rejected'` and `meta.reason='concrete_action_too_vague'`.

## Component 4 — Deduplicator

**What it does:** Before each INSERT, checks `task_queue` for an existing open proposal with a
matching `fingerprint`.

**Query:**
```sql
SELECT id, metadata->>'recurrence_count' as recurrence_count
FROM task_queue
WHERE status IN ('queued', 'claimed')
  AND metadata->>'proposal_fingerprint' = $1
  AND metadata->>'task_type_label' = 'improvement_proposal'
LIMIT 1
```

**If match found (Q3 — resolved 2026-04-24):** Do BOTH:

**Step 1** — UPDATE the existing row to track recurrence:

```sql
UPDATE task_queue
SET metadata = metadata || jsonb_build_object(
  'recurrence_count', (metadata->>'recurrence_count')::int + 1,
  'last_seen_chunk_id', $chunk_id,
  'severity', $new_severity,           -- recalculated based on new count
  'needs_root_cause_review', CASE WHEN (metadata->>'recurrence_count')::int + 1 >= 2
                                   THEN true ELSE false END
)
WHERE id = $existing_id
```

**Step 2** — INSERT a new `task_queue` row for the current chunk with `meta.recurrence_of = <existing_id>`.
   This ensures the current chunk's proposal is visible to coordinator's pickup loop even when an
   older row for the same pattern already exists. The `recurrence_of` link surfaces the history.

**`recurrence_count >= 2` flag:** When the existing row's count reaches 2 after the UPDATE,
set `metadata.needs_root_cause_review = true`. This signals to coordinator: "this fix isn't
sticking — the root cause may be deeper than the proposed action."

**If no match:** proceed to Component 5 (Queuer) with no UPDATE.

**Acceptance:** Running the engine twice on the same completed chunk: (1) existing row has
`recurrence_count` incremented by 1, AND (2) one new row inserted with `meta.recurrence_of`
pointing to the existing row. Total new rows per re-run = 1 (not 0, not 2).

**Note on `task_queue` schema:** The existing `task_queue.metadata` column is `jsonb`. The
Deduplicator uses `metadata->>'proposal_fingerprint'` and `metadata->>'task_type_label'` as
discriminators. No migration required — these are new keys in an existing jsonb column.
Builder must verify the current `task_queue` schema before assuming this is safe (Check-Before-Build).

## Component 5 — Queuer

**What it does:** INSERTs new proposals into `task_queue`.

**INSERT shape:**

```typescript
{
  task: `Improvement proposal: ${proposal.concrete_action.slice(0, 80)}`,
  description: proposal.concrete_action,
  priority: proposal.severity === 'blocking' ? 1
           : proposal.severity === 'meaningful' ? 2
           : 3,
  status: 'queued',
  source: 'improvement_engine',
  metadata: {
    task_type_label: 'improvement_proposal',
    category: proposal.category,
    severity: proposal.severity,
    concrete_action: proposal.concrete_action,
    engine_signal: proposal.engine_signal,
    measurement: proposal.measurement,
    source_chunk_id: proposal.source_chunk_id,
    proposal_fingerprint: proposal.fingerprint,
    recurrence_count: 0,
    f17_compliance: proposal.engine_signal,  // F17 explicit field
    f18_compliance: proposal.measurement     // F18 explicit field
  }
}
```

**Acceptance:** Queued rows appear in `task_queue` with `status='queued'` and are retrievable
by coordinator's next pickup cycle (pickup loop already queries `task_queue WHERE status='queued'`
— no changes needed to pickup logic). Verify: after engine runs on a chunk with mismatches,
`SELECT * FROM task_queue WHERE metadata->>'task_type_label' = 'improvement_proposal' AND
status = 'queued'` returns ≥1 row.

## Component 6 — Notifier

**What it does:** Sends one Telegram message per chunk via `outbound_notifications` if ≥1
proposal was queued.

**INSERT into `outbound_notifications`:**

```typescript
{
  channel: 'telegram',
  payload: {
    text: buildNotificationText(proposals),
    parse_mode: 'Markdown',
    reply_markup: {
      inline_keyboard: [[
        { text: 'Approve all', callback_data: `improve_approve_all:${chunk_id}` },
        { text: 'Review each', callback_data: `improve_review:${chunk_id}` },
        { text: 'Dismiss', callback_data: `improve_dismiss:${chunk_id}` }
      ]]
    }
  },
  correlation_id: task_queue_id,
  requires_response: true
}
```

**Message format:**

```
[LepiOS Improvement Engine] Chunk {chunk_id}
{N} proposal(s) queued

1. [{category}] {concrete_action — truncated to 120 chars}
   Severity: {severity} | Recurrence: #{recurrence_count}
2. ...
```

**Acceptance:** Colin receives one Telegram message per chunk that had proposals queued. Not
one message per proposal. If 0 proposals were queued (clean chunk), no message is sent.
The existing notifications-drain cron picks up the `outbound_notifications` row and delivers it
via the Telegram proxy endpoint (same as all other outbound notifications).

**Callback routing note:** The `[Approve all]` / `[Review each]` / `[Dismiss]` callbacks are
handled by the existing Telegram webhook (`app/api/telegram/webhook/route.ts`). Builder must
add dispatch logic for `improve_approve_all:*`, `improve_review:*`, `improve_dismiss:*` callback
prefixes. On `Approve all`: mark all proposals for this chunk_id as `status='approved'` in
task_queue. On `Dismiss`: mark all as `status='dismissed'`. On `Review each`: send individual
proposal messages with per-proposal approve/dismiss buttons.

## Component 7 — Auto-Proceed Gate (Q5 — resolved 2026-04-24)

**What it does:** Before sending a proposal to the Notifier (which queues a Telegram approval
request), checks whether the proposal qualifies for automatic proceeding without Colin sign-off.

**All five criteria must hold for auto-proceed:**

1. `category` is one of `{tooling, code_pattern, test_coverage}`
2. `severity === 'nice_to_have'`
3. Fully reversible by `git revert` — no new files, no schema changes, no env vars added
4. Existing test suite passes after the change (verified by running tests in CI, not assumed)
5. `action_signature` matches a row in `auto_proceed_patterns` with `approval_count >= 3` and `enabled = true`

**If all 5 hold:** mark proposal as `status='auto_proceeded'` in task_queue, skip Notifier,
log `agent_events` with `action='improvement_engine.auto_proceeded'`.

**If any criterion fails:** route to Notifier for Colin sign-off via Telegram thumbs. This is
the default path — auto-proceed is the exception, not the rule.

**`auto_proceed_patterns` table (new — migration required):**

```sql
CREATE TABLE auto_proceed_patterns (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  category      text NOT NULL,
  action_signature text NOT NULL,  -- normalized first 80 chars of concrete_action
  approval_count  int NOT NULL DEFAULT 0,
  last_approved_at timestamptz,
  enabled       boolean NOT NULL DEFAULT false,
  created_at    timestamptz NOT NULL DEFAULT now()
);
```

A pattern row is created the first time Colin approves a proposal of that category+action_signature.
`approval_count` increments on each subsequent approval. `enabled` flips to `true` only when
`approval_count >= 3`. Builder must verify this table does not already exist before migrating.

**Acceptance:** A proposal with `category='tooling'`, `severity='nice_to_have'`, fully
reversible, and a matching `auto_proceed_patterns` row with `approval_count >= 3` is marked
`status='auto_proceeded'` in task_queue without a Telegram message sent. All other proposals
generate a Telegram message.

## Kill Signals

1. **Engine fires but produces 0 proposals for a chunk with documented grounding mismatches** →
   Analyzer is not reading the audit trail correctly. Stop; do not ship silently-failing engine.

2. **Duplicate proposals in task_queue after second engine run** → Deduplicator fingerprint
   matching is broken. Stop; fix before proceeding.

3. **Telegram message contains raw JSON** → Notifier is not formatting proposals correctly.
   Stop; fix before shipping (Colin reads on mobile — JSON messages are unusable).

4. **Engine adds columns to task_queue** → This is a schema migration. Builder must stop,
   flag the migration, and escalate per ARCHITECTURE.md §3 rule 3.

5. **Engine is invoked inline with task pickup (synchronous)** → This blocks the pickup loop.
   Engine must be async / fire-and-forget.
