## Scope

One sentence: Build a thin attribution layer that records which agent, run, coordinator session,
and commit triggered every entity write, stores it in a queryable `entity_attribution` table, and
exposes a `/api/attribution/[entity_type]/[entity_id]` endpoint so the cockpit can display
provenance for any row.

**One acceptance criterion:** After the component ships, every `task_queue` INSERT triggered
by the improvement-engine, pickup-cron task claim, and coordinator fire includes an
`entity_attribution` row. The query endpoint returns those rows correctly. F18 coverage
metric shows ≥90% of agent-triggered task_queue writes are attributed.

---

## Out of Scope (v1)

- Retroactive backfill of pre-ship writes (NULL = "before attribution shipped")
- Attribution columns on `deals`, `bets`, `scan_results` (no active agent write path today —
  add when those writes become agent-driven)
- UI panel in cockpit (log the data first; surface it once there are enough rows to display)
- Attribution for human-triggered writes (Colin's manual actions via Telegram thumbs)
- Git blame integration (commit_sha is captured at deploy-gate; linking to entity rows is v2)
- RLS policy on entity_attribution (read-only endpoint is fine; no user-facing PII)

**Why deferred:** All deferred items need the table to exist first. Ship the table and writer;
extend coverage incrementally as write paths are added.

## Component 1 — Schema (Migration)

**Migration file:** `supabase/migrations/0020_add_entity_attribution.sql`

```sql
CREATE TABLE entity_attribution (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  entity_type           TEXT NOT NULL,          -- 'task_queue', 'improvement_proposal', etc.
  entity_id             UUID NOT NULL,           -- FK to the entity's primary key
  action                TEXT NOT NULL,           -- 'created', 'claimed', 'completed', 'approved', 'dismissed', 'auto_proceeded'
  actor_type            TEXT NOT NULL,           -- 'improvement_engine' | 'coordinator' | 'task_pickup_cron' | 'human' | 'cron'
  actor_id              TEXT,                    -- specific identifier (e.g., coordinator session name)
  run_id                UUID,                    -- task-pickup cron run UUID (from task_queue.claimed_by)
  coordinator_session_id TEXT,                   -- Anthropic Routines session ID
  source_task_id        UUID,                    -- task_queue.id that triggered this write (if any)
  commit_sha            TEXT,                    -- from deploy-gate; nullable
  occurred_at           TIMESTAMPTZ NOT NULL DEFAULT now(),
  details               JSONB                    -- what changed: fingerprint, concrete_action, etc.
);

-- Query patterns: changelog for one entity, all actions by one actor
CREATE INDEX idx_entity_attribution_entity
  ON entity_attribution (entity_type, entity_id, occurred_at DESC);

CREATE INDEX idx_entity_attribution_actor
  ON entity_attribution (actor_type, occurred_at DESC);

CREATE INDEX idx_entity_attribution_run
  ON entity_attribution (run_id) WHERE run_id IS NOT NULL;

CREATE INDEX idx_entity_attribution_session
  ON entity_attribution (coordinator_session_id) WHERE coordinator_session_id IS NOT NULL;

COMMENT ON TABLE entity_attribution IS
  'Provenance log: which agent/run/session/commit created or modified each entity.';
```

Builder must list current migration files before creating this one.

**Acceptance:** Migration applies cleanly (`supabase db push` or via Supabase MCP). Table

## Component 3 — Attribution Writer

**File:** `lib/attribution/writer.ts` (new)

```typescript
import { createClient } from '@/lib/supabase/server'
import type { AttributionContext } from './types'

export async function recordAttribution(
  context: AttributionContext,
  entity: { type: string; id: string },
  action: string,
  details?: Record<string, unknown>
): Promise<void>
```

**Behavior:**

- Inserts one row into `entity_attribution` using the provided context + entity + action
- Fire-and-forget: the caller does NOT await success; attribution failure must never
  block the primary write. Wrap the insert in a try/catch; log errors to `agent_events`
  with `action='attribution.write_failed'`, `status='error'` — do not throw.
- No batching in v1 (single insert per call)

**Attribution failure policy (important):**

```typescript
try {
  await supabase.from('entity_attribution').insert({ ... })
} catch (err) {
  // attribution failure is non-fatal — log and continue
  await logEngineEvent({ action: 'attribution.write_failed', status: 'error', ... })
}
```

**Acceptance:** Unit test confirms: (a) successful write produces one row in entity_attribution
with correct fields, (b) a simulated Supabase error does NOT cause the writer to throw —
caller receives undefined, not an exception.

## Component 4 — Wire Improvement Engine

**File:** `lib/harness/improvement-engine.ts` — beef up

**Where to call `recordAttribution`:**

1. **When a proposal is queued** (after successful INSERT to task_queue):

   ```typescript
   // After: await supabase.from('task_queue').insert(proposalRow)
   await recordAttribution(
     {
       actor_type: 'improvement_engine',
       source_task_id: triggering_task_queue_id,
       run_id: triggeringClaimedBy ?? undefined, // task_queue.claimed_by of the completed task
     },
     { type: 'task_queue', id: newProposalId },
     'created',
     { category: proposal.category, fingerprint: proposal.fingerprint, severity: proposal.severity }
   )
   ```

2. **When a proposal is auto-proceeded** (Component 7 path):
   ```typescript
   await recordAttribution(
     { actor_type: 'improvement_engine', source_task_id: ..., run_id: ... },
     { type: 'task_queue', id: proposalId },
     'auto_proceeded',
     { reason: 'auto_proceed_pattern_match', approval_count: matchedPattern.approval_count }
   )
   ```

**Context reconstruction (no context passed in — engine reads it):**
The improvement-engine fires as its own cron, so it doesn't receive the pickup run_id.
It reconstructs context from the completed task_queue row:

- `source_task_id` = the completed task's `id`
- `run_id` = the completed task's `claimed_by` (the pickup cron's UUID that ran the task)
- `coordinator_session_id` = query `agent_events` WHERE
  `meta->>'task_id' = <task_id>` AND `action = 'coordinator.session_started'`
  → take the first result's `meta->>'coordinator_session_id'` if present; else null

Builder: read the current improvement-engine.ts queuer block (approx. line 606–619) before
editing. Do NOT change the proposal insert shape — add the recordAttribution call after it.

**Acceptance:** After engine processes one completed chunk, query:

```sql
SELECT * FROM entity_attribution
WHERE entity_type = 'task_queue' AND action = 'created'
ORDER BY occurred_at DESC LIMIT 5;
```

Confirm: rows exist with `actor_type='improvement_engine'`, `source_task_id` populated,
`details` contains `fingerprint`.

## Measurement (F18)

```
module: attribution
metric_name: Attribution coverage rate
unit: % of agent-triggered task_queue INSERT operations with a corresponding entity_attribution row
capture_method: |
  Numerator: COUNT(*) FROM entity_attribution WHERE entity_type = 'task_queue'
    AND occurred_at > <ship_date>
  Denominator: COUNT(*) FROM task_queue WHERE source IN ('improvement_engine', 'task_pickup_cron', 'coordinator')
    AND created_at > <ship_date>
  Coverage = numerator / denominator
benchmark:
  type: colin-target
  value: ≥90% coverage within 7 days of ship (allows for edge cases and error paths)
  rationale: 100% is aspirational but edge-case writes (retries, partial failures) may
             skip attribution if the primary write fails before attribution fires
baseline: 0% (no attribution rows exist before this component ships)
surfacing_path: |
  1. morning_digest field: "attribution: N records today, X% coverage"
  2. Query: SELECT actor_type, COUNT(*) FROM entity_attribution
       WHERE occurred_at > now() - interval '24 hours'
       GROUP BY actor_type
     → tells Colin "who wrote what yesterday"
alert_threshold: |
  Coverage drops below 50% for 3 consecutive days → silent failure somewhere in the write chain
  → Telegram alert via agent_events monitor
```

**Secondary metrics (query available immediately after ship):**

- Rows per actor_type (breakdown: how much work each agent is doing)
- Most active source_task_id (which tasks spawn the most downstream writes)
- Coordinator sessions with zero attributed writes (coordinator fired but nothing recorded)

## Engine-Feeding (F17)

| Signal                                 | Path to engine                                                                                                       |
| -------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| actor_type distribution                | Base rate of agent vs human vs cron activity — feeds path probability engine's workload model                        |
| source_task_id chains                  | Task lineage graph: proposal → claim → coordinator fire — reveals which proposals generate real work                 |
| coordinator_session_id → entity writes | Correlates coordinator quality (how many entities a session creates) with eventual grounding outcomes                |
| attribution.write_failed events        | Reliability signal: if write failures cluster on a specific actor, that actor's write path has a reliability problem |

Attribution's primary F17 contribution: it makes every other agent's signal traceable.
Without provenance, two agent_events rows can't be confidently linked. With attribution,
the improvement-engine can ask "did this coordinator session's work get grounded?" and get
a real answer.

## Kill Signals

1. **Attribution writer throws and propagates** — the writer MUST NOT throw. If it does,
   every agent write path that calls it will start failing. Stop; fix the try/catch.

2. **Migration conflicts with existing number** — do not overwrite an existing migration.
   Check `ls supabase/migrations/` and use the next sequential number.

3. **Zero attribution rows after 24h of agent activity** — writer is silently failing.
   Check agent_events for `attribution.write_failed` events. Stop; diagnose before reporting done.

4. **Query route returns attribution for wrong entity** — index or query bug. Stop; data
   integrity matters more than shipping speed.

5. **AttributionContext imported from a route handler file** — violates F11 (leaks server-only
   modules into client bundle). Types must live in `lib/attribution/types.ts` with zero
   runtime imports.
