# Follow-up: Twin Retrieval Gap (2026-05-01)

The `/api/twin/ask` endpoint is live and reachable — `POST https://lepios-one.vercel.app/api/twin/ask` returns 200. The previous "unreachable" reports in the auto-proceed-log were written from the coordinator's cloud sandbox process, which cannot make outbound HTTP calls to the production URL (H1-B sandbox isolation), not from a failed deploy. The route itself was never down. However, a real functional gap remains: coordinator-style questions consistently return `retrieval_path: 'none'` and `escalate_reason: 'insufficient_context'`, meaning both vector search (Ollama embed, blocked when tunnel is down) and FTS return zero matching chunks. The root cause is the F-L14 registry-driven ingest gap — `ingest-claude-md.ts` maintains a hard-coded entity list that does not include newer arch rules and coordinator principles, so those knowledge chunks were never written to the `knowledge` table and are invisible to the twin. Cache-match was re-enabled despite this gap (2026-05-01, both Sprint 4 and Sprint 5 blocks in `docs/sprint-state.md`) because coordinator judgment serves as the fallback when twin returns insufficient_context — the autonomous flow is unblocked even with a degraded twin. Full resolution gates on the F-L14 task landing: `ingest-claude-md.ts` must derive its entity list from `lib/rules/registry.ts` at runtime so new rule entries are automatically ingested on the next nightly run.
