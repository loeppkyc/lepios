## Chunk list (refined from brief seeds A–E)

### Chunk A — SP-API Integration + Today Live / Yesterday

**One-sentence scope:** Wire SP-API Orders endpoint into LepiOS and display today's orders, revenue, estimated payout (from SP-API Finances, no hardcoded rate), and units — side-by-side today and yesterday panels.

**Acceptance criterion:** Today Live panel and Yesterday panel display order count, revenue (pre-fees), estimated net settlement amount (from SP-API Finances balance proration, not `revenue * 0.65`), and units sold. Numbers match Seller Central Orders page to the penny on Colin's spot-check.

**Out of scope:** Recent Days table (Chunk C), fee backfill, any hardcoded payout rate, any Streamlit 65% approximation.

**Grounding checkpoint:** Colin pulls up Seller Central on his phone, checks today's orders/revenue against LepiOS Today Live. Penny-match required. Estimated payout must come from the Finances endpoint, not a formula.

---

### Chunk B — What You're Owed (SP-API Finances)

**One-sentence scope:** Display Amazon's pending settlement balance, FBA inventory unit count, and average cost per unit — sourced from SP-API Finances and Inventory endpoints, with a manual override path preserved.

**Acceptance criterion:** "What You're Owed" panel shows (a) pending settlement dollar figure matching Seller Central Payments → Statement View to the penny, (b) FBA unit count, (c) average cost per unit (from cost data if available; blank/manual-override if not). No fabricated numbers.

**Out of scope:** Fee projections, life P&L, financial snapshot, anything not directly from SP-API Finances or Inventory endpoints.

**Phase 2 lock required:** The "average cost per unit" source must be decided during acceptance doc authorship — builder cannot start Chunk B with this open. Phase 2 must pick one: (a) blank if SP-API returns no cost basis, (b) manual-override UI, or (c) env-var default. Coordinator will escalate this decision to Colin if SP-API Check-Before-Build does not resolve it.

**Grounding checkpoint:** Colin checks Seller Central Payments → Statement View for pending settlement balance. Must match LepiOS What You're Owed to the penny.

**Dependency:** Chunk A must be complete (SP-API credentials and client are wired). Chunk B reuses the SP-API client Chunk A establishes; it cannot be built independently.

---

### Chunk C — Recent Days Table

**One-sentence scope:** Render a last-10-days sales table from SP-API Orders with honest zeros — if fee or payout data is unavailable for a day, the cell shows zero or blank (never a back-filled estimate), with column labels that state their data source explicitly.

**Acceptance criterion:** Table renders 10 rows; each row shows date, order count, revenue, units; fee/payout cells are blank or zero when SP-API does not return data for that day (no `revenue * 0.35` backfill); column headers include source attribution ("SP-API Orders").

**Out of scope:** Fee detail per order, monthly roll-ups, Sales Chart (Sprint 9), view toggles (Sprint 8).

**Grounding checkpoint:** Colin spot-checks 3 of the last 10 days against Seller Central Business Reports. Numbers match. Zeros show as zeros (not estimates).

**Dependency:** Chunk A (SP-API Orders client and auth). Chunk B not required — Chunk C does not touch Finances endpoint.

---

### Chunk D — Statement Coverage Grid

**One-sentence scope:** Build the per-account × per-month statement coverage grid by reading the Dropbox statement folder listing via the Dropbox API (and/or n8n webhook) — green cell if statement file is present, red if absent, no fabrication.

**Acceptance criterion:** Grid renders with one row per account and one column per month (trailing 12 months minimum); cells are green (file detected) or red (no file); grid matches Colin's Dropbox statement folder exactly on manual spot-check count.

**Out of scope:** Statement content parsing, PDF extraction, financial totals from statements. This is presence/absence only.

**Grounding checkpoint:** Colin opens his Dropbox statement folder, counts present months per account, compares to the LepiOS grid. Must match exactly.

**Dependency:** No dependency on Chunks B or C. Dropbox API credentials must exist (Check-Before-Build: verify existing Dropbox integration in repo before proposing new auth). Can be built in parallel with B and C after the Chunk A grounding checkpoint passes (coordinator needs that pass before greenlighting Chunk D, to avoid burning tokens on D if A surfaces a sprint-direction problem).

---

### Chunk E — Business Review Page Shell

**One-sentence scope:** Create the `/business-review` Next.js page that composes Chunks A, B, C, D with a refresh control, last-updated timestamp, and honest loading/error states for each section independently.

**Acceptance criterion:** Page loads at project's `/business-review` route; each of the four sections renders independently (one section's API error does not blank the whole page); refresh control triggers re-fetch; last-updated timestamp reflects actual fetch time; no silent error swallowing (error states are visible, not suppressed).

**Out of scope:** Navigation, design polish beyond functional shell, any deferred sections (Life P&L, Financial Snapshot, Inventory Potential Profit, Sales Chart, AI Briefing, view toggles).

**Grounding checkpoint:** Full page load with real data → all four sections render with real data or honest empty states. Colin confirms no section is silently blank when it should show data.

**Dependency:** Chunks A, B, C, D must all be complete (Chunk E composes their components). Chunk E is strictly last.

## Chunk ordering rationale

Per CHUNK-ORDERING principle (colin-principles.md): dependency first, then grounding-confidence descent.

1. **Chunk A first** — it is the foundation dependency for B and C, and it is the highest-uncertainty integration (SP-API Orders endpoint live-test required per Principle 1 before the acceptance doc is written). Its failure cheaply signals if the sprint direction is wrong. Grounding-heavy, new terrain.

2. **Chunk B second** — depends on A (SP-API client), uses Finances endpoint (second SP-API surface, additive risk). Can only proceed after A grounding-checkpoint passes.

3. **Chunk C third** — depends on A (same SP-API Orders client), but does not depend on B. Could technically parallel with B after A passes. Ordering here is: B before C because What You're Owed has higher monetary stakes and earlier grounding risk than the Recent Days table.

4. **Chunk D parallel-eligible after A** — independent of B and C (Dropbox is a separate integration from SP-API). D does not wait for B or C to finish. D does wait for A's grounding-checkpoint pass: if A reveals a sprint-direction problem, Chunk D hasn't burned tokens on a potentially abandoned sprint. Coordinator will greenlight Chunk D build in parallel with B and C once the Chunk A grounding checkpoint passes.

5. **Chunk E last** — requires A, B, C, D complete. Shell integration work only makes sense when all four components exist.
