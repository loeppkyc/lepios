## Scope

Wire the SP-API Orders endpoint into LepiOS and display, side by side, a Today panel and a Yesterday panel — each showing order count, revenue (pre-fees), and units sold. The payout/settlement field on both Today and Yesterday panels renders as a static label: **"Full payout estimate in Sprint 5"** — not a number. Rationale: the Finances endpoint returns a total pending balance, not a per-day contribution; displaying that number under a Today or Yesterday heading would be a misleading label (Principle 6). Honest empty state now; real per-order fee data in Sprint 5.

**Order state definition (locked):** LepiOS counts **confirmed orders only** — `OrderStatuses: Unshipped,PartiallyShipped,Shipped,Canceled,Pending`. The API call fetches all statuses in one request; the route handler splits them client-side: confirmed (Unshipped + PartiallyShipped + Shipped + Canceled) drives the count/revenue/units numbers. Pending drives the indicator only. Rationale: Pending revenue is excluded from the headline number (Principle 6 — Pending orders can be canceled, so including them overstates confirmed income). But suppressing the Pending count entirely hides information when confirmed = 0 and Pending > 0.

**Pending indicator requirement:** When confirmed order count = 0 and pending count > 0, the Today panel must display a small secondary label: `"(N pending not shown)"` alongside the zero count. This prevents the panel from being indistinguishable from "no sales today" when all sales are still in Pending state. The indicator is display-only — it does not contribute to revenue, units, or any number.

**Revenue definition (locked):** Revenue = sum of `ItemPrice.Amount` across all items in confirmed orders — **pre-tax, pre-fee item price only**. `OrderTotal.Amount` (which includes tax) must NOT be summed — this was the source of the $4.20 grounding failure (buyer in BC, 12% provincial tax). Requires `GET /orders/v0/orders/{orderId}/orderItems` per confirmed order to retrieve `ItemPrice.Amount`.

**Tax capture (locked — v0 infrastructure for LepiOS tax/GST module):** Tax = sum of `ItemTax.Amount` + `ShippingTax.Amount` across all items in confirmed orders. Tax is NOT decorative — it is intentionally plumbed through `DayPanelData.taxCad` so the future LepiOS income tax + GST module can pull daily tax totals without re-querying SP-API. Display: revenue is the primary value; tax appears as a secondary sub-line (e.g. "+ $4.20 tax"). Do not aggregate tax for Pending orders — consistent with revenue exclusion. `ShippingPrice.Amount` and `PromotionDiscount.Amount` are excluded from both revenue and tax.

Rationale for pre-tax revenue: (a) SC displays item price pre-tax as the default "Order total" — penny-match is achievable against that number; (b) tax is collected for CRA and remitted, never earned by the seller; (c) consistent with how Finances (Chunk B) treats settlement amounts.

**One acceptance criterion:** Today and Yesterday panels display confirmed order count, revenue (pre-tax `ItemPrice.Amount`, confirmed orders only) with tax as a sub-line, and units sold — and the revenue numbers match the **confirmed-only view in Seller Central** (Order total, pre-tax, Pending excluded) to the penny on Colin's spot-check. When confirmed = 0 and pending > 0, the panel shows "(N pending not shown)" indicator. The payout field on both panels renders as the static label "Full payout estimate in Sprint 5" with no number shown. Both `revenueCad` and `taxCad` are stored in `DayPanelData` for downstream use by the tax/GST module.

**Full reconciliation (confirmed 2026-04-22):**

```
API OriginalTotal (Group 3):      $928.17  CAD  ← financialEventGroups endpoint
Account Level Reserve (SC stmt):  − $96.23  CAD  ← SC Payments → Net Proceeds page
                                  ─────────────
SC Net Proceeds (Statement View):  $831.94  CAD  ← $928.17 − $96.23 = $831.94 ✓ EXACT

SC Total Balance (Payments dash):  $917.83  CAD  ← intermediate; gap from API = $10.34

  Interpretation: SC builds the reserve incrementally as orders settle. Total Balance
  reflects partial reserve application; Net Proceeds reflects full reserve. The $10.34
  is SC's own internal reserve application lag — not an API timing issue, not a
  definition mismatch. It resolves as orders in the period settle.
```

**Hypothesis B confirmed.** `OriginalTotal` is gross of reserve. SC Net Proceeds is the honest "what will be paid" number.

**Which SC view to use for Chunk B grounding:** SC Payments → Statement View → **Net Proceeds** ($831.94), NOT Total Balance ($917.83). Net Proceeds = OriginalTotal − full reserve = what Amazon will actually transfer. Total Balance is an intermediate calculation that changes as the reserve applies; it has no clean API equivalent and should not be used as a grounding target.

**Reserve availability in the API:** The reserve ($96.23) is NOT present in the `financialEventGroups` response. Group 3's payload has only 5 fields (`FinancialEventGroupId`, `ProcessingStatus`, `OriginalTotal`, `BeginningBalance`, `FinancialEventGroupStart`). ## Implementation notes (builder must read)

These are not kill signals but required constraints discovered via live-test. Violating either produces incorrect data without a visible error.

**Constraint 0 — Order status filter (locked definition):**
Use `OrderStatuses: Unshipped,PartiallyShipped,Shipped,Canceled` — exactly this list, no additions. Pending is excluded by design. Do not add Pending without a new sprint scope decision. This filter applies to both Today and Yesterday queries.

**Constraint 1 — `CreatedBefore` future-date rule (Orders endpoint):**
The SP-API Orders endpoint returns HTTP 400 if `CreatedBefore` is a future timestamp. For "today" queries, use `CreatedAfter` only (today at midnight Edmonton time) and omit `CreatedBefore` entirely. The Streamlit reference passes both `CreatedAfter` and `CreatedBefore` — that pattern is valid for Yesterday and historical day queries (both boundaries are in the past), but NOT for Today. Builder must branch on this.

**Constraint 2 — CAD currency filter (Finances endpoint — Chunk B note):**
The Finances endpoint returns groups across ALL Amazon marketplaces (CA + MX + others). At least one open group has `CurrencyCode: MXN` with $0 amount. When `finances.ts` is built in Chunk B, it must filter `OriginalTotal.CurrencyCode === 'CAD'` before summing pending balance. Cross-currency sum produces a meaningless number. This constraint is documented here so Chunk B builder doesn't rediscover it from scratch.
