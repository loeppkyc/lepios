## Coverage Rule — Gmail Arrival → Covered Month

**Rule:** a `gmail_statement_arrivals` row with `arrival_date` in month M means month M−1
is covered for that account.

- Arrival in April 2026 → March 2026 is covered (consistent with Colin's Q1 example:
  "Bonvoy statement arrives ~Apr 20 = March statement complete")
- Arrival in January 2026 → December 2025 is covered (cross-year; falls in 2025 band)
- Current Edmonton month is always 'pending' under this rule (no email arrives for current
  month until next month, when the statement closes)

**If `statement_period_end` is non-null** (future gmail-scanner upgrade populates this):
use that date's year+month as the covered month instead of arrival_month − 1. Route must
check for non-null `statement_period_end` first, fall back to arrival_month − 1 if null.
This makes D v2 forward-compatible with the period-extraction upgrade.

**Known limitation:** some accounts issue statements mid-month (statement period ends April
20, email arrives April 22 = arrival in April but covered period includes April). In these
cases, arrival_month − 1 (March) would be wrong — April should be green. Manual override
(filed_override) is the mitigation until `statement_period_end` is populated. Document this
in the component as a footer note.

### Gmail coverage query

const all2025 = get2025Months()

### Coverage map build

```typescript
// Build a Set<string> of "classifierName:YYYY-MM" for covered months
const coveredSet = new Set<string>()
for (const row of arrivals ?? []) {
  let coveredYear: number
  let coveredMonth: number

  if (row.statement_period_end) {
    // statement_period_end is a date string "YYYY-MM-DD"
    const [y, m] = row.statement_period_end.split('-').map(Number)
    coveredYear = y
    coveredMonth = m
  } else {
    // Fall back to arrival_month - 1
    const [y, m] = row.arrival_date.split('-').map(Number)
    const prev = previousMonth(y, m)
    coveredYear = prev.year
    coveredMonth = prev.month
  }
  const key = `${row.account_name}:${coveredYear}-${String(coveredMonth).padStart(2, '0')}`
  coveredSet.add(key)
}
```

### Per-account coverage map (same NO_ACTIVITY and override logic as v1)

```typescript
const { year: nowYear, month: nowMonth } = currentEdmontonDate()
// Load overrides (fail-open — same try/catch pattern as v1)
const manualOverrides = new Set<string>()
try {
  const { data: rows } = await gate.supabase
    .from('statement_coverage_overrides')
    .select('account_key, year_month')
  for (const row of rows ?? []) {
    manualOverrides.add(`${row.account_key}:${row.year_month}`)
  }
} catch { /* non-fatal */ }

const accounts = BUSINESS_ACCOUNTS.map((account) => {
  const coverage: Record<string, CoverageStatus> = {}
  const noActivityMonths = NO_ACTIVITY[account.key] ?? []

  for (const month of allMonths) {
    if (noActivityMonths.includes(month)) {
      coverage[month] = 'no_activity'
    } else if (coveredSet.has(`${account.classifierName}:${month}`)) {
      coverage[month] = 'filed'
    } else if (manualOverrides.has(`${account.key}:${month}`)) {
      coverage[month] = 'filed_override'
    } else {
      const [y, m] = month.split('-').map(Number)
      coverage[month] = cellStatus(y, m, nowYear, nowMonth)
    }
  }

  return { key: account.key, label: account.label, coverage }
})
```

## Migration — RLS Policy for gmail_statement_arrivals

**Finding:** `gmail_statement_arrivals` has `rowsecurity = true` but NO policies defined.
In Supabase, enabled RLS with no policies = deny all. The route's `gate.supabase` (user
auth client) returns 0 rows today. Must add SELECT policy before D v2 route can read the table.

**Pattern match:** `statement_coverage_overrides` uses `has_business_access()` for ALL.
Same function applies here — `gmail_statement_arrivals` is a business-review supporting
table read by the same route.

**Migration number:** claim next available from `.claude/migration-claims.json` at build time.
Builder runs `node scripts/next-migration-number.mjs` to get the number.

```sql
-- Migration NNNN_gmail_statement_arrivals_rls.sql
ALTER TABLE gmail_statement_arrivals ENABLE ROW LEVEL SECURITY;  -- already enabled; idempotent

CREATE POLICY gmail_statement_arrivals_business_read
  ON gmail_statement_arrivals
  FOR SELECT
  USING (has_business_access());
```

**Note for builder:** This migration modifies an existing table's RLS policies. Per
`.claude/CLAUDE.md` seam rules, the commit must include `[seam-approved]` in the message.
