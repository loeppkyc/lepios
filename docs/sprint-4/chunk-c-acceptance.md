## Implementation notes (builder must read)

**Constraint C-1 — Caching is mandatory**
Use `export const revalidate = 900` on the `/api/business-review/recent-days` route (15-minute server-side cache). Do NOT use `force-dynamic`. Historical days (completed past-midnight-Edmonton days) will never change — their order data is final. Without caching, 10 days × N orderItems calls renders on every page load, making the table unacceptably slow. The response must include `fetchedAt: string` (ISO timestamp of actual fetch) so the component can display "Last updated: X min ago." The component reads `fetchedAt` from the response — same pattern as `WhatYouOwePanel` Constraint B-8.

**Constraint C-2 — Inline types only in the component (critical)**
The Chunk B bug that prevented `WhatYouOwePanel` from mounting: `import type` from route handler files causes Turbopack to traverse the full module graph, leaking server-only `crypto` into the client bundle, breaking mount silently. Define response types inline in `RecentDaysTable.tsx` — do NOT import from `route.ts`. Example:

```typescript
// Inline — do NOT import from route file
interface RecentDayRow {
  date: string // ISO date string "2026-04-22"
  orders: number
  revenueCad: number
  units: number
  fetchedAt: string
}
interface RecentDaysResponse {
  rows: RecentDayRow[]
  fetchedAt: string
}
```

**Constraint C-3 — Revenue method is `ItemPrice.Amount`, not `OrderTotal.Amount`**
Must use the same per-order `/orderItems` fetch pattern as the Today/Yesterday panel. `OrderTotal.Amount` includes provincial tax (grounding failure from Chunk A: $4.20 BC tax at 12%). `ItemPrice.Amount` is the correct pre-tax revenue field. This is more API calls but it is the only path to a penny-match.

**Constraint C-4 — Date window construction**
Generate 10 day windows using the existing `dayBoundaryUTC(date, 'start' | 'end')` function from `lib/amazon/orders.ts`. Day 1 = yesterday; day 10 = 10 days ago. Each window: `createdAfter = dayBoundaryUTC(d, 'start')`, `createdBefore = dayBoundaryUTC(d, 'end')`. Both boundaries are in the past for all 10 days — the `CreatedBefore future-date` constraint (Chunk A Constraint 1) does not apply to fully-past windows. Do NOT include today (partial day — covered by TodayYesterdayPanel).

**Constraint C-5 — Fetch strategy to avoid rate limits**
With 10 day windows + N orderItems calls, naive parallel fetching risks SP-API rate limiting. Recommended: fetch the 10 order-list requests sequentially (to respect the Orders endpoint rate limit), collect all confirmed order IDs across all days, then fetch all orderItems in parallel (as per today-yesterday route's `buildFinanceMap` pattern). The 15-minute cache (Constraint C-1) means this expensive fetch only runs once per cache period.

**Constraint C-6 — Pending orders shown as sub-line, not primary**
Primary numbers (Orders, Revenue, Units) are confirmed-only (Unshipped + PartiallyShipped + Shipped + Canceled) — the numbers that matter for tax and payout. Pending orders on the same creation date are surfaced as a muted sub-line beneath the primary value: "+ $X.XX pending" in Revenue and "+ N pending" in Units. Sub-line is hidden (not rendered) when pendingRevenueCad/pendingUnits = 0 for that day. This matches the "(X pending not shown)" pattern on the Today panel and resolves the SC mismatch: confirmed + pending together account for the SC Business Report total. Historical Pending is real (net-30 B2B orders can stay Pending > 10 days) — suppressing it silently caused the grounding failure.

**Constraint C-7 — Zero display**
Days with zero confirmed orders display `0` for order count and units, and `$0.00` for revenue. Do NOT display blank or `—` for these fields — zero is the honest value (we know there were no confirmed orders, not that we failed to fetch them). The `—` is reserved for Fees and Net (data that was never available from this endpoint).

**Constraint C-8 — Fees and Net columns**
Both columns render `—` for every row. Column headers label the source: suggested labels are **"Fees (Sprint 5)"** and **"Net (Sprint 5)"** so the placeholder is self-explanatory. No formula, no estimation, no `revenue * 0.35` anywhere. Rationale: consistent with the pattern established in Chunks A and B; makes explicit to Colin that fee data is intentionally absent.
